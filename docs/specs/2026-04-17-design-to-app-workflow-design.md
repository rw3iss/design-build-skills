# Design-to-App Workflow — Spec

**Date:** 2026-04-17
**Author:** ryan@vendidit.com
**Status:** Draft — pending review

## Purpose

Turn a natural-language design brief into (a) a set of generated design images via Midjourney on Discord, and (b) on request, a Preact + SCSS scaffold that matches those images as closely as possible. Both halves are invokable independently: image generation without build, or build from an existing folder of images.

## Scope

**In scope:**
- A `designer` skill that takes a prompt, generates images via Midjourney (Discord bot), and stores them in a per-request folder.
- An `upscale` sub-operation of `designer` that takes specific previously-generated images and returns Midjourney U1-U4 upscales.
- A `design-build` skill that consumes a folder of images (plus optional extra prompt) and emits a Preact + TypeScript + SCSS scaffold under `app/`.
- A shared TypeScript library for Discord client, storage, and config.
- A manual fallback mode that prints `/imagine <prompt>` for the user to run themselves.
- A distributable install script (`install.sh`) that pulls the skills from a configurable remote Git repository and installs them into `~/.claude/skills/`.

**Out of scope (v1):**
- Generating images from non-Midjourney providers. Provider abstraction is sketched in the interfaces but only Midjourney is implemented.
- Upscaling arbitrary images not generated by our workflow (would need Real-ESRGAN/Topaz).
- Producing backend code or tests. The `design-build` skill generates UI only.
- An always-on bot daemon. Bot is ephemeral per invocation. A `--persist` upgrade is a future iteration.

## User flow

### Path 1 — Design, then build
1. User: "generate some designs for a checkout page with glassmorphism aesthetic"
2. Claude invokes `designer`. Skill walks up from CWD to find `DESIGN.md`, crafts prepared prompt, calls Midjourney bot, saves images.
3. Skill replies with indexed list: `1: designs/checkout-glass/images/01.png ... 4: .../04.png`.
4. User: "build from 2 and 4, make the buttons rounder"
5. Claude invokes `design-build` with images `[02.png, 04.png]` and extra note "buttons rounder".
6. Skill walks up for `BUILD.md`, writes `PLAN.md`, scaffolds Preact app under `designs/checkout-glass/app/`.

### Path 2 — Build from an existing folder
1. User: "build a login form from the images in ./my-designs/login/"
2. Claude invokes `design-build` with an explicit images path; skill derives `<request-name>` from the folder name (or a prompted name), writes outputs to `designs/<request-name>/app/`.

### Path 3 — Upscale, then build
1. After a design run, user: "upscale 2 and 4, then build from the upscales"
2. Claude invokes `designer upscale <request-name> 2 4` — bot clicks U2/U4 on the stored message ID, downloads, saves as `02-up.png`/`04-up.png`.
3. User: "build from the upscales"
4. Claude invokes `design-build` referencing `02-up.png`/`04-up.png`.

### Path 4 — Manual fallback
1. User: "generate designs for X, manual mode"
2. Skill prepares the prompt, prints `/imagine <prompt>` and the target path, tells user to paste into Discord and drop the result image(s) into `designs/<request-name>/raw/`.
3. User does that; runs a follow-up: "process raw images for checkout-glass"
4. Skill splits any 2×2 grids it finds in `raw/` into `images/01..04.png`.

## Architecture

### Directory layout (user-facing)

Output is written into the invoking project's CWD:

```
<project-cwd>/designs/<request-name>/
  prompts/
    original.md          verbatim user input
    prompt.md            prepared Midjourney-ready prompt
    build-notes.md       appended at build time, if any
  raw/
    mj-<msgid>.png       original grid attachment from Discord
    mj-<msgid>.json      { prompt, message_id, channel_id, timestamp, u_button_custom_ids }
  images/
    01.png 02.png 03.png 04.png        split quadrants
    02-up.png 04-up.png                upscales, if requested
  app/
    PLAN.md              generated implementation plan
    README.md            scaffold README with architecture notes
    package.json
    config/                            tsconfig, vite, oxlint, dev scripts
    src/
      app/                             shell, routing, entry point
      components/                      UI components built from images
      services/                        service-layer interfaces + adapters
        api/                           ApiClient interface + MockApiAdapter (default)
      mock/
        data/                          static JSON fixtures (users.json, products.json, ...)
        MockApiAdapter.ts              implements ApiClient using data/ + small delay
        mockServer.ts                  optional: in-memory request/response "server" if mutable state matters
    styles/                            global SCSS framework (variables, mixins, utilities)
```

### Skill layout (install-side)

Both skills live under `~/.claude/skills/` as portable, global skills:

```
~/.claude/skills/
  designer/
    SKILL.md
    scripts/
      mj_bot.ts          ephemeral bot: one operation per invocation
      prepare_prompt.ts  merges user prompt + aesthetic + DESIGN.md
      split_grid.ts      cuts 2×2 PNG into four quadrants
      upscale.ts         drives U1–U4 clicks on a stored message_id
      setup_check.ts     verifies env, connects, checks channel + MJ presence
      process_raw.ts     processes images in raw/ (manual-mode ingestion)
    lib/
      discord_client.ts  gateway + interaction-component plumbing
      midjourney.ts      MJ-specific command payloads and button parsing
      storage.ts         request-name resolver, path helpers
      config.ts          env + ~/.config/designer/config loader
      prompt_prep.ts     DESIGN.md walk-up + prompt assembly
    package.json         discord.js, sharp, zod, tsx, dotenv
    tsconfig.json
    README.md            setup guide

  design-build/
    SKILL.md
    scripts/
      build_plan.ts      emits PLAN.md from images + prompts + BUILD.md
      scaffold_preact.ts writes config/, src/, styles/, package.json
      select_images.ts   resolves user-facing selections to file paths
    package.json
    tsconfig.json
    README.md
```

`design-build` imports shared utilities from `../designer/lib/` via a relative path pinned in a small `shared.ts` shim — avoids publishing a package, keeps both skills installable by copying two sibling directories into `~/.claude/skills/`. Installation dependency is documented: installing `design-build` without `designer` is unsupported. If `designer` is absent, `design-build`'s `shared.ts` throws a clear error at import.

### Component boundaries

| Component | Purpose | Depends on |
|---|---|---|
| `designer/SKILL.md` | Skill entry point — documents the commands Claude uses | `scripts/*` |
| `designer/scripts/*` | One binary per verb. Each reads JSON from argv, writes JSON to stdout. Callable standalone from Claude's Bash tool. | `lib/*` |
| `designer/lib/discord_client.ts` | Thin wrapper over `discord.js` for login, message fetch, interaction-component click, attachment download | `discord.js` |
| `designer/lib/midjourney.ts` | MJ-specific knowledge: application ID, `/imagine` command ID, button custom_id → U/V/R parsing | `discord_client.ts` |
| `designer/lib/storage.ts` | Request-name kebab-casing, directory creation, metadata JSON read/write | — |
| `designer/lib/prompt_prep.ts` | Walk-up for `DESIGN.md`, merge rules, produce final `prompt.md` text | `storage.ts` |
| `design-build/scripts/build_plan.ts` | Walk-up for `BUILD.md`, synthesize plan from images + original prompt + extra notes | `../designer/lib/*` |
| `design-build/scripts/scaffold_preact.ts` | Emit Preact + SCSS skeleton. Does NOT generate component bodies from images — that step is Claude-driven. | template files in `design-build/templates/` |

### Bot lifecycle

Ephemeral. Every operation is a single invocation:

```
node --import tsx ~/.claude/skills/designer/scripts/mj_bot.ts generate '{"prompt":"...","request":"...","count":1,"upscale":false}'
```

1. Bot loads `.env` / config, logs in via token, waits for `ready`.
2. Sends `/imagine` as a slash-command interaction targeting Midjourney's application ID in the configured channel.
3. Awaits a bot message from Midjourney whose embed/content references the invoking user (MJ prefixes its replies with `@<user>`), with attachments.
4. Downloads the attachment, persists `raw/mj-<msgid>.png` + `raw/mj-<msgid>.json` (preserving `message.components[]` custom_ids for later upscales).
5. Runs `split_grid.ts` to write `images/01..04.png`.
6. If `upscale: true`, also clicks U1–U4 and collects each reply.
7. Prints `{"status":"ok","request":"...","images":[...]}` to stdout, exits.

Timeouts: 300s for the initial MJ completion, 180s per upscale, each configurable. On timeout, exits non-zero with a structured error; Claude surfaces it and suggests `--manual` mode.

### Manual mode

`--manual` flag on the design command:
- Skips bot spawn entirely.
- Prints: `/imagine <prompt>` and `Drop the resulting image(s) into designs/<request-name>/raw/ then say "process raw images for <request-name>".`
- A subsequent `process_raw.ts` call scans `raw/*.png`, detects 2×2 grids (by aspect ratio + rough dimension match), splits each into `images/NN.png`, and reports back.

### Configuration

Two-tier:

1. **`~/.config/designer/config.json`** (user-global, persists across projects):
   ```json
   {
     "discordBotToken": "Bot ...",
     "discordChannelId": "...",
     "discordGuildId": "...",
     "midjourneyAppId": "<discovered on first setup run>",
     "midjourneyImagineCommandId": "<discovered on first setup run>"
   }
   ```

   Both Midjourney identifiers are resolved at first `setup_check` run by fetching the channel's application commands list — no hardcoded IDs in the codebase.
2. **Env-var overrides**: `DESIGNER_DISCORD_TOKEN`, etc. Useful for one-off runs or CI.

Never checked into any repo. `setup_check.ts` creates the file on first run (prompting for token via stderr → user-pasted arg).

### DESIGN.md / BUILD.md lookup

Walk up from CWD toward filesystem root; first file named `DESIGN.md` (resp. `BUILD.md`) found at any level wins. Stops at `/` or at the first `.git` directory boundary (so a repo-root `DESIGN.md` is found but searches don't escape into a parent monorepo by accident). Overridable:
- `designer --design-md <path>`
- `design-build --build-md <path>`

If found, its contents are prefixed into the prepared prompt under a `# Project rules` section. If not found, the skill prints a one-line notice but proceeds.

### Prompt preparation

`prompt_prep.ts` produces `prompts/prompt.md` by combining:

1. User's original brief (verbatim, stored as `prompts/original.md`).
2. Extracted "aesthetic/style" vs "engineering requirements" sections — detected by headings like `# Design / Aesthetic` vs `# Engineering / Requirements`. Falls back to treating the whole thing as mixed guidance.
3. `DESIGN.md` contents as project rules.
4. Midjourney-specific flags appended at the end: `--ar 3:2` (default), `--v 6.1`, `--style raw` unless user overrides.

The final `/imagine` prompt text is the design/aesthetic section + concise functional hints, with all MJ flags at the end. Engineering-only content is NOT sent to Midjourney (it's UI noise there) — it's preserved for the build phase.

### Selection UX

Design skill output ends with a brief list:

```
Generated 4 images in designs/checkout-glass/images/:
  1: 01.png   2: 02.png   3: 03.png   4: 04.png

Reply "build from [numbers]" to invoke design-build.
```

Claude parses the user's follow-up ("build from 2 and 4, make the buttons rounder") into `imagesArg=[2,4]` and `extraPrompt="make the buttons rounder"` and invokes `design-build`. No AskUserQuestion gate.

## Data flow

```
user prompt → designer CLI
  → prepare_prompt.ts ──→ prompts/original.md + prompts/prompt.md
  → mj_bot.ts (generate) ──→ raw/mj-<id>.png + .json
  → split_grid.ts ──→ images/01..04.png
  → stdout: { request, images }
  ... user picks images ...
user follow-up → design-build CLI (imagesArg, extraPrompt)
  → build_plan.ts → app/PLAN.md (includes entity list inferred from images for mock fixtures)
  → scaffold_preact.ts → app/package.json, config/, src/, styles/, src/services/api/, src/mock/
  → Claude fills in: component bodies, ApiClient interface methods, mock/data/*.json fixtures
  → stdout: { app_dir }
```

## Error modes and handling

| Failure | Surface | Remediation |
|---|---|---|
| No config file | `setup_check.ts` runs on first `designer` call; prints exactly which env vars or config fields are missing | User edits config / runs setup-check directly |
| Bad token | Login error from discord.js | Token re-prompt; point to discord.com/developers/applications |
| Bot not in channel | Explicit check in setup_check (channel fetch returns nothing) | Instructions to invite bot with required scopes |
| Midjourney bot not in channel | setup_check verifies by fetching channel members / recent messages | Instructions to add Midjourney and subscribe |
| MJ timeout | Bot exits non-zero with `{status:"timeout", elapsed}` | Skill suggests `--manual` mode |
| Grid split fails (wrong aspect) | `split_grid.ts` refuses, reports dimensions | Skill surfaces, user runs with `--no-split` or uses raw file directly |
| Upscale on missing message_id | `upscale.ts` reports that the original raw JSON is absent | User must re-generate |
| `BUILD.md` missing | Notice only, not an error | Proceeds with generic build rules |

## Installation & distribution

Both skills, their shared library, their `package.json`s, and their templates live inside the `@vendidit/tools` monorepo at `packages/design-and-build-skills/`. An `install.sh` at the *package* root handles copying the skill subtrees into `~/.claude/skills/` and bootstrapping npm dependencies. Users never need to clone or install the rest of the monorepo — the installer uses a shallow + sparse Git checkout to pull only the package path.

### Repository layout (inside `Vendidit/tools`)

```
Vendidit/tools/ (GitHub repo)
  … other packages …
  packages/
    design-and-build-skills/             (this package)
      install.sh                         bash installer (main entry)
      uninstall.sh                       removes the two skill dirs, preserves config
      update.sh                          alias for `install.sh --update`
      README.md                          install instructions + quick-start
      CLAUDE.md                          session-resume context for Claude Code
      package.json                       monorepo workspace entry; private
      VERSION                            semver string, copied into installed skill dirs
      docs/
        specs/…                          design docs
      skills/
        designer/
          SKILL.md
          scripts/ … lib/ … package.json … tsconfig.json … README.md
        design-build/
          SKILL.md
          scripts/ … templates/ … package.json … tsconfig.json … README.md
```

Repo URL + package path are recorded in the installer as three constants near the top:
```bash
REPO_URL="${DESIGNER_SKILLS_REPO:-https://github.com/Vendidit/tools.git}"
REPO_REF="${DESIGNER_SKILLS_REF:-main}"
PACKAGE_PATH="packages/design-and-build-skills"
```
All three overridable via env vars for forks / private mirrors / pinned releases.

### `install.sh` behavior

**Contract:** idempotent, safe to re-run, never destroys user config.

Steps the script performs:

1. **Preflight.** Verify `git` (≥2.25, for sparse-checkout), `node` (≥22), `npm`, `rsync` present. Fail fast with actionable messages.
2. **Fetch source.** Shallow + sparse clone of `REPO_URL` at `REPO_REF` into `~/.cache/vendidit-design-and-build-skills/`, scoped to `PACKAGE_PATH`. On re-run, does `git -C ... fetch && git -C ... checkout <ref>` instead of re-cloning. Concretely:
   ```bash
   git clone --depth 1 --filter=blob:none --sparse --branch "$REPO_REF" "$REPO_URL" "$CACHE"
   git -C "$CACHE" sparse-checkout set "$PACKAGE_PATH"
   ```
   Honors `--ref <tag-or-sha>` to override `REPO_REF`.
3. **Copy skills.** For each of `designer` and `design-build`:
   - Remove the existing `~/.claude/skills/<name>/node_modules/` (stale deps are the #1 failure mode on re-install).
   - `rsync -a --delete --exclude node_modules --exclude .env $CACHE/$PACKAGE_PATH/skills/<name>/ ~/.claude/skills/<name>/`
4. **Install deps.** `cd ~/.claude/skills/<name> && npm ci` for each (or `npm install` if no lockfile).
5. **Stamp version.** Copy `$CACHE/$PACKAGE_PATH/VERSION` into each installed skill dir.
6. **Config preservation.** Never touch `~/.config/designer/config.json` or any `.env` already on disk.
7. **Post-install check.** If `~/.config/designer/config.json` is missing, print the first-run setup instructions. If it exists, print a short "installed, ready to use" line plus the command to re-run setup_check if Discord config needs refreshing.

**Flags:**
- `--update` — same as normal run, but refuses to proceed if the cache has uncommitted changes (helps when users have been hand-editing the cache for development).
- `--ref <sha|tag|branch>` — pin to a specific revision; overrides `REPO_REF`.
- `--skill <designer|design-build>` — install or update just one skill, skip the other.
- `--local <path>` — skip the clone and copy from a local path. The path may be either the package root (contains `install.sh`) or the monorepo root (contains `packages/design-and-build-skills/`); the script auto-detects.
- `--dry-run` — print what would happen, don't touch anything.

**Invocation patterns supported:**

```bash
# One-liner: fetch install.sh straight from the package path in the monorepo and pipe to bash
curl -fsSL https://raw.githubusercontent.com/Vendidit/tools/main/packages/design-and-build-skills/install.sh | bash

# Explicit clone of just this package (sparse), for users who want to read the script first
git clone --depth 1 --filter=blob:none --sparse https://github.com/Vendidit/tools.git
cd tools && git sparse-checkout set packages/design-and-build-skills
cd packages/design-and-build-skills && ./install.sh

# Full clone (simpler, heavier — fine for dev)
git clone https://github.com/Vendidit/tools.git
cd tools/packages/design-and-build-skills && ./install.sh

# Developing the skills themselves against your local working copy
cd <wherever>/tools/packages/design-and-build-skills
./install.sh --local .

# Pin a release
./install.sh --ref design-and-build-skills-v0.2.0
```

### `uninstall.sh` behavior

Removes `~/.claude/skills/designer/` and `~/.claude/skills/design-build/`. Asks before removing `~/.config/designer/config.json` (never silently drops credentials). Does NOT remove `~/.cache/vendidit-design-and-build-skills/` unless `--purge` is passed.

### Update path

- `./install.sh --update` (or the shortcut `./update.sh`) — fetches latest from `REPO_URL@REPO_REF` into the cache, re-runs copy + deps.
- `install.sh` is the single mechanism; there's no second "updater" binary to maintain.

### Versioning

Tags follow the monorepo's per-package convention `design-and-build-skills-vX.Y.Z` (so tags from other packages in `Vendidit/tools` don't collide). The package root contains a `VERSION` file; `install.sh` copies it into each installed skill directory so `setup_check.ts` can report the installed version and a future update nag can compare against the remote.

### From-a-chat install

Because the install path is a single shell command, a user can ask Claude "install the design skills" and Claude runs the one-liner via the Bash tool. No slash command needed; the install script is the interface.

## Setup (captured in the skill's README)

Runs *after* `install.sh` has placed the skills on disk. Discord setup is a one-time user task; the installer cannot automate it.

1. Discord app: discord.com/developers/applications → New Application → Bot tab. Enable "Message Content Intent".
2. Generate bot token, save to `~/.config/designer/config.json`.
3. OAuth2 URL generator: scopes `bot` + `applications.commands`. Permissions: Read Messages, Read Message History, Use Application Commands, Send Messages.
4. Create (or reuse) a Discord server you own. Subscribe to Midjourney; invite the Midjourney bot.
5. Invite your custom bot to the same server with the OAuth URL.
6. Run `node --import tsx ~/.claude/skills/designer/scripts/setup_check.ts` — bot connects, resolves Midjourney's application + command IDs, writes them to the config, verifies all required permissions. A small `designer-setup` wrapper shell script in the skill root can alias this.

## SOLID/style principles for the generated Preact app (emitted into `BUILD.md` context)

Summary — full requirements copied verbatim from the user's brief into `design-build/scripts/build_plan.ts`'s template:

- TypeScript + Preact 10 + Vite
- Custom SCSS framework, no external UI libs; `.mobile.scss` companions for mobile media queries
- `config/` folder hosts tsconfig, vite, oxlint, dev scripts, hot reload
- Services abstracted via adapter classes so an API backend can be swapped in
- All bindings hoisted to component top for easy integration replacement
- Multiple render variants per UI component (button styles, form styles, etc.)
- No backend code; no test suite

### Mock data layer (mandatory for every build)

Every scaffolded app ships with a working mock backend so the design is demonstrable end-to-end without any real API. Architecture:

1. **`services/api/ApiClient.ts`** — a TypeScript interface describing every method the UI calls (`getProducts()`, `createOrder(order)`, etc.). The UI depends only on this interface.
2. **`mock/MockApiAdapter.ts`** — default implementation that satisfies `ApiClient`. Reads from `mock/data/*.json`, simulates latency (~100–400 ms configurable), supports pagination/filtering/sorting over the static fixtures. If a flow requires mutable state (add to cart, submit form), `MockApiAdapter` holds it in-memory for the session.
3. **`mock/data/*.json`** — fixtures generated per build. The shape of the fixtures is derived from what the design actually shows (e.g., if an image depicts a product grid with price + rating + image, the fixture has `id/title/price/rating/imageUrl` rows). Realistic sample data, not lorem ipsum, ~10–30 rows per entity so lists and empty states both feel natural.
4. **`services/api/index.ts`** — single injection point exposing `const apiClient: ApiClient = new MockApiAdapter(...)`. This is the line a downstream integrator replaces with a real HTTP adapter (`new HttpApiAdapter(baseUrl, authToken)`). Components never import `MockApiAdapter` directly.
5. **`mock/mockServer.ts`** (optional, emitted only when needed) — a thin in-memory "server" object used by `MockApiAdapter` when the design involves cross-call mutation (e.g., cart, wishlist, multi-step checkout). Keeps state centralized and resettable.

Hot-reload note: mock data changes in `mock/data/*.json` are picked up by Vite's HMR, so tweaking fixtures during the design review loop is instant.

Swap-out documented in `app/README.md`: a short "Integrating a real backend" section shows the one-file change (`services/api/index.ts`) and a skeleton `HttpApiAdapter` that mirrors `ApiClient`.

## Non-goals (explicit)

- Do NOT auto-invoke `design-build` from `designer`. User always says "build it" in a follow-up.
- Do NOT attempt to log the user into Discord with credentials. Bot token only.
- Do NOT ship an always-on bot service in v1.
- Do NOT generate component bodies via template files. Division of labor: `scaffold_preact.ts` writes the *deterministic* parts (package.json, tsconfig, vite config, SCSS variable scaffolding, app shell entry points, empty `services/api/` and `mock/` folder structure with placeholder files) from fixed templates. The *design-specific* parts — component JSX, per-component SCSS matching the image, `ApiClient` method signatures, mock fixture contents — are written by Claude *after* the scaffold lands, using the images + PLAN.md as its brief. This keeps the script deterministic and testable while letting Claude do the creative part it's good at.

- The "no backend" non-goal means no real server code, no persistence, no networked APIs. It does NOT exclude the mandatory in-browser mock layer described above — that's treated as part of the UI, not a backend.

## Open questions resolved inline

- Language: TypeScript (aligned with user's stack; switched from initial Python pick).
- Directory: flat (`app/` directly under request root; no `generated/` wrapper).
- Skill split: two skills + shared `lib/`.
- Selection: conversational.
- Bot: ephemeral.
- DESIGN.md/BUILD.md: walk-up from CWD.
- Upscale: MJ-native path only (existing messages we generated).

## Test plan (after implementation)

- **Installer dry-run**: `./install.sh --dry-run` on a clean machine prints the full plan without touching disk.
- **Fresh install**: `./install.sh` on a clean machine produces working `~/.claude/skills/designer/` and `~/.claude/skills/design-build/` with `node_modules` populated; `setup_check.ts` runnable.
- **Idempotent re-install**: running `./install.sh` twice leaves identical filesystem state (no duplicates, no clobbered config).
- **Update**: commit a trivial change to the repo, run `./install.sh --update`, verify change reflected in installed copy and `node_modules` rebuilt only if lockfile changed.
- **Local dev install**: `./install.sh --local .` from a working checkout installs from disk, skipping `git pull`.
- **Uninstall**: `./uninstall.sh` removes skill dirs, prompts before touching config; `--purge` also clears the cache.
- `designer setup` fresh: walks through the checklist, fails with clear remediation at each missing piece.
- `designer generate "simple test prompt" --manual`: verifies prompt prep + path creation without touching Discord.
- `designer generate "..."` end-to-end: real Discord call, check grid image in raw/, 4 quadrants in images/.
- `designer upscale <req> 2`: click U2 on stored message, verify `02-up.png`.
- `design-build --images-from designs/<req>/images --select 2,4 --extra "..."`: PLAN.md generated, scaffold under app/, `npm run dev` in the scaffold starts Vite with hot reload, and the rendered UI displays data pulled from `src/mock/data/*.json` via `MockApiAdapter`.
- Mock swap-out smoke test: replace `services/api/index.ts`'s `new MockApiAdapter(...)` with a stub `HttpApiAdapter` that throws on every call — verify compile succeeds and UI surfaces errors cleanly, proving the injection seam works.
- `design-build --images-from ./my-designs/foo` (standalone): request name derived, outputs under `designs/foo/app/`.
